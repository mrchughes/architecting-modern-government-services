# Strategy: LLM-Assisted Development in Microservice Architectures

**A Framework for Efficient, Robust AI-Assisted Development at Scale**

*For Solution Architects*

Version 1.0  
January 2026

## Executive Summary

### The Challenge

Large Language Models (LLMs) promise to accelerate software development, but without proper controls they create chaos in complex microservice architectures. At scale, uncontrolled LLM assistance leads to:

- **Boundary violations**: Services become coupled, architectural integrity degrades
- **Cost explosion**: Token usage spirals to unsustainable levels (£500k-£1M+ annually)
- **Quality degradation**: Too much context creates noise, LLMs make poor decisions

Traditional controls—prompts, code review, and process—fail because they operate after the damage is done.

### The Solution

This strategy presents a **three-tier architectural model** that makes LLM assistance both **efficient** (95%+ token cost reduction) and **robust** (boundary violations prevented by design):

1. **Domain Architecture Tier** - Strategic boundaries (DDD bounded contexts)
2. **Microservice Architecture Tier** - Service boundaries (contracts, APIs, events)
3. **Clean Architecture Tier** - Internal boundaries (domain/application/infrastructure layers)
4. **Infrastructure & Deployment Tier** - Platform boundaries (IaC, pipelines, configs)

Each tier uses **Service-Scoped RAG** (Retrieval-Augmented Generation) to provide focused context, combined with **CI/CD guardrails** to enforce boundaries automatically.

### The Outcome

**Efficiency Gains:**
- **Token cost reduction**: 95-98% (from £1M+/year to £20-30k/year for 100 services)
- **Conversation continuity**: 20+ turn conversations vs. 3-5 without RAG
- **Developer velocity**: 30-40% reduction in repetitive work

**Robustness Guarantees:**
- **Zero boundary violations**: Multi-layered defense prevents architectural drift
- **Automated enforcement**: CI/CD validates every change, no manual review needed
- **Safe scaling**: Linear growth to 100+ services without process overhead

### For Solution Architects

This is not "LLM tooling"—this is **architectural enforcement through intelligent context management**.

RAG provides the mechanism. Contract testing provides the verification. Architecture is the outcome.

## 1. The Problem - Why LLMs Break Things at Scale

### 1.1 The Uncontrolled LLM Scenario

Consider a typical development request:

```
Developer: "Fix the refund bug in the payments service"
```

**Without architectural controls**, here's what happens:

```
LLM Behavior:
├─ Loads entire monorepo (2,000+ files, ~2M tokens)
├─ Sees: payments, notifications, identity, finance, all shared libs
├─ Context cost: ~£0.24 per request (£0.12/1M input tokens)
├─ "Helpfully" refactors across 6 services
├─ Creates new coupling between payments and notifications
├─ Breaks contracts in 3 places
├─ Changes shared library used by 12 services
└─ Result: Architectural chaos requiring days of cleanup
```

**The conversation history problem compounds this:**

```
Turn 1: Load 2M tokens (£0.24)
Turn 2: 2M history + 2M reload = 4M tokens (£0.48)
Turn 3: 4M history + 2M reload = 6M tokens (£0.72)
Turn 4: 6M history + 2M reload = 8M tokens (£0.96)
Turn 5: Context window limit reached, must reset

Total for 5 turns: £2.40
Usable turns: 4-5 maximum before conversation breaks
```

**At scale**, this becomes catastrophic:

```
Annual Cost (100 services, 10 changes per service per month):
= 100 services × 10 changes × 12 months
= 12,000 changes per year
= 12,000 × 4 turns average × £0.60/turn
= £28,800/year in token costs alone

Hidden costs:
+ Architectural cleanup: ~20% of changes break boundaries = £150k/year
+ Integration fixes: ~15% break contracts = £100k/year
+ Review overhead: All changes need deep review = £200k/year
Total: ~£479k/year for 100 services
```

### 1.2 Three Critical Failure Modes

**Failure Mode 1: Boundary Violations**

Without enforcement, LLMs don't respect architectural boundaries:

- **Service boundaries**: Changes leak across services, creating coupling
- **Domain boundaries**: Business logic appears in wrong contexts
- **Layer boundaries**: Infrastructure concerns pollute domain logic
- **Contract boundaries**: APIs change without consumer notification

**Example violation:**
```typescript
// LLM "helpfully" adds this to payments service
import { sendEmail } from '../notifications/emailService';

// Now payments is coupled to notifications implementation
// Should use event: refund.processed → notifications subscribes
```

**Failure Mode 2: Cost Explosion**

Token costs grow exponentially, not linearly:

| Services | Avg Files | Avg Tokens | Cost/Request | Annual Cost (120 changes/service) |
|----------|-----------|------------|--------------|-----------------------------------|
| 10 | 300 | 300k | £0.036 | £5,184 |
| 50 | 1,500 | 1.5M | £0.18 | £129,600 |
| 100 | 3,000 | 3M | £0.36 | £518,400 |
| 200 | 6,000 | 6M | £0.72 | £2,073,600 |

*Note: Costs based on Claude Sonnet 4 pricing (£0.12/1M input tokens, £0.60/1M output tokens), assuming 4 turns per change*

The problem: **Loading everything on every request** makes conversation history explode.

**Failure Mode 3: Quality Degradation**

More context ≠ better results. Too much noise reduces accuracy:

| Approach | Context Size | Signal-to-Noise | Accuracy | Files Changed Correctly |
|----------|--------------|-----------------|----------|-------------------------|
| Full repo | 2M tokens | 0.5% | 45-60% | Wrong file 40% of time |
| Full service | 400k tokens | 6% | 65-75% | Wrong file 25% of time |
| Service RAG | 25k tokens | 95%+ | 85-95% | Correct file 90%+ of time |

**Why noise degrades quality:**
- LLM distracted by irrelevant patterns
- Business rules buried in noise
- "Helpful" refactoring touches unrelated code
- Test coverage missed because tests not prioritized

### 1.3 Why Traditional Approaches Fail

**Approach 1: "Just prompt the LLM carefully"**

```
Prompt: "Only change files in services/payments/. 
Do not touch other services. Follow Clean Architecture."
```

**Reality**: LLMs are helpful, not obedient
- Instructions ignored when LLM sees "obvious improvements"
- Cannot enforce what LLM can see
- Visibility = capability (if it can see it, it will change it)

**Approach 2: "Rely on code review"**

Problems:
- Review happens **after** code is written (waste)
- Reviewers must understand all touched services (doesn't scale)
- Subtle coupling hard to spot in review
- 50+ services = review becomes bottleneck

**Approach 3: "Only give LLM access to one service folder"**

Better, but still problematic:
- Still loads 100-150 files (400k tokens)
- Conversation history still bloats
- No prioritization (all files equal weight)
- Cost: £0.048/request vs £0.003 with RAG (16x more expensive)

### 1.4 What's Actually Needed

**Architectural enforcement, not process enforcement:**

1. **Pre-emptive boundaries**: LLM cannot see what it shouldn't change
2. **Focused retrieval**: Only relevant context loaded (95% noise elimination)
3. **Automated verification**: CI/CD enforces rules, not humans
4. **Economic sustainability**: Token costs stay linear, not exponential

The next sections describe how to achieve this.

## 2. The Solution - Three-Tier Segregation with RAG

### 2.1 The Core Principle

> **Enforce boundaries through architecture, optimize with RAG, verify with CI/CD**

Traditional approach:
```
LLM → Change Code → Human Review → (maybe) Catch Problems
```

This approach:
```
LLM → RAG (focused context) → Propose Change → CI/CD Guards → Auto-verify → Merge or Block
```

**Key insight**: Make violations **impossible**, not **discouraged**.

### 2.2 The Four-Tier Architectural Model

Each tier represents a different level of architectural concern, with its own:
- **Sphere of influence** (what it controls)
- **RAG scope** (what context is provided)
- **Verification mechanism** (how correctness is enforced)
- **Ownership** (who governs changes)

```
┌──────────────────────────────────────────────────────────┐
│ TIER 1: DOMAIN ARCHITECTURE                              │
│ Sphere: Business capabilities, bounded contexts          │
│ RAG: Business processes, policies, domain rules          │
│ Verify: Domain rule tests                                │
│ Owner: Enterprise/Solution Architects                    │
├──────────────────────────────────────────────────────────┤
│ TIER 2: MICROSERVICE ARCHITECTURE                        │
│ Sphere: Service boundaries, contracts, APIs              │
│ RAG: OpenAPI specs, event schemas, service registry      │
│ Verify: Contract tests (consumer-driven)                 │
│ Owner: Solution Architects + Service Leads               │
├──────────────────────────────────────────────────────────┤
│ TIER 3: CLEAN ARCHITECTURE (Internal Service)            │
│ Sphere: Domain/Application/Infrastructure layers         │
│ RAG: Service code, tests, ADRs, patterns                 │
│ Verify: Unit tests, integration tests, TDD               │
│ Owner: Service Development Teams                         │
├──────────────────────────────────────────────────────────┤
│ TIER 4: INFRASTRUCTURE & DEPLOYMENT                      │
│ Sphere: Platforms, IaC, pipelines, configurations        │
│ RAG: Terraform, Helm charts, pipeline definitions        │
│ Verify: Infrastructure contract tests                    │
│ Owner: Platform/DevOps Team                              │
└──────────────────────────────────────────────────────────┘
```

### 2.3 How Tiers Work Together

**The tiers are nested and coordinated:**

```
Domain Rules (Tier 1)
    ↓ (constrain)
Service Contracts (Tier 2)
    ↓ (implemented by)
Clean Architecture (Tier 3)
    ↓ (deployed via)
Infrastructure (Tier 4)
```

**Example flow:**

1. **Domain tier** defines: "Refunds must be auditable and atomic with payments"
2. **Microservice tier** specifies: `POST /refunds` API contract, `refund.processed` event schema
3. **Clean Architecture tier** implements: Domain logic respecting business rules, clean layers
4. **Infrastructure tier** deploys: Kubernetes deployment, monitoring, audit logging

**Changes flow with integrity:**
- Cannot change service contract without domain validation
- Cannot implement logic that violates domain rules (tests fail)
- Cannot deploy without infrastructure contracts satisfied

### 2.4 The Two Phases: Design and Evolution

**Critical insight**: Each tier goes through the same two phases:

**Design Phase** (Initial creation)
- **RAG scope**: BROAD (exploration, discovery)
- **LLM role**: Propose, suggest, generate options
- **Human role**: Decide, approve, freeze decisions
- **Goal**: Establish boundaries and contracts

**Evolution Phase** (Ongoing changes)
- **RAG scope**: NARROW (enforcement, precision)
- **LLM role**: Implement within boundaries
- **Human role**: Spot-check, approve high-risk changes
- **Goal**: Maintain integrity, prevent drift

```
       Design Phase              Evolution Phase
       (Broad RAG)               (Narrow RAG)
            ↓                          ↓
    ┌──────────────┐          ┌──────────────┐
    │  Exploration │   →      │  Refinement  │
    │  Discovery   │   freeze │  Maintenance │
    │  Proposals   │  ──────→ │  Enforcement │
    └──────────────┘          └──────────────┘
```

### 2.5 Control Mechanisms at Each Tier

**Three layers of defense at every tier:**

1. **RAG Scoping** (Pre-emptive)
   - Only relevant context loaded
   - LLM physically cannot see what it shouldn't change
   - 95% of problems prevented here

2. **Contract/Test Verification** (Automatic)
   - Domain rule tests, contract tests, unit tests, infrastructure tests
   - Breaking changes fail automatically
   - No human judgment required

3. **CI/CD Guardrails** (Enforcement)
   - Path validation (changed files must be in allowed scope)
   - Dependency checking (only allowed imports)
   - Architectural fitness functions (coupling metrics, layer violations)

**The result**: Violations are **impossible**, not just **discouraged**.

## 3. How RAG Makes This Efficient

### 3.1 What is Service RAG?

**Service RAG** (Service-Scoped Retrieval-Augmented Generation) is a pattern where an LLM retrieves context **only** from code, docs, and contracts belonging to a single architectural scope—and nothing else.

**Not RAG:**
```
Developer: "Fix refund bug"
System: Dumps entire service folder into LLM context
        - 150 files
        - 400,000 tokens
        - £0.048 per request
        - Everything is equally weighted
```

**Service RAG:**
```
Developer: "Fix refund bug"
System: Queries service RAG index with "refund bug"
        - Retrieves 8-12 most relevant files:
          ✓ refund.test.ts (business rules - highest priority)
          ✓ RefundService.ts (domain logic)
          ✓ ProcessRefundUseCase.ts (application layer)
          ✓ refund-settlement-adr.md (why it works this way)
        - 25,000 tokens
        - £0.003 per request
        - Signal-to-noise ratio: 95%+
```

### 3.2 The Three Critical RAG Benefits

**Benefit 1: Focus (Signal-to-Noise Ratio)**

| Approach | Files Seen | Relevant Files | Signal-to-Noise | LLM Accuracy |
|----------|------------|----------------|-----------------|--------------|
| Full repo | 2,000+ | 8-12 | 0.5% | 45-60% |
| Full service folder | 150 | 8-12 | 6% | 65-75% |
| Service RAG | 8-12 | 8-12 | 95%+ | 85-95% |

**Why focus improves accuracy:**
- Tests appear first (encode ground truth behavior)
- ADRs provide "why" context (not just "what")
- LLM not distracted by irrelevant patterns
- Consistent retrieval = predictable behavior

**Benefit 2: Cost Efficiency (Token Economics)**

**Single request comparison:**

```
Task: "Add validation to refund amount"

Without RAG (full service folder):
- Input tokens: 400,000
- Input cost: 400k × £0.12/1M = £0.048
- Output tokens: ~2,000
- Output cost: 2k × £0.60/1M = £0.0012
- Total: £0.0492 per request

With Service RAG:
- Input tokens: 25,000
- Input cost: 25k × £0.12/1M = £0.003
- Output tokens: ~2,000
- Output cost: 2k × £0.60/1M = £0.0012
- Total: £0.0042 per request

Savings: 88% per request
```

**Conversation over time (THE CRITICAL DIFFERENCE):**

```
Without RAG:
Turn 1: 400k tokens → £0.048
Turn 2: 400k (history) + 400k (reload) = 800k → £0.096
Turn 3: 800k + 400k = 1.2M → £0.144
Turn 4: 1.2M + 400k = 1.6M → £0.192
Turn 5: Context window approaching limit

Total: £0.48 for 5 turns
Conversation must reset every 4-5 turns
Developer experience: Frustrating, loses context

With Service RAG:
Turn 1: 25k tokens → £0.003
Turn 2: 25k (history) + 20k (new retrieval) = 45k → £0.0054
Turn 3: 45k + 18k = 63k → £0.0075
Turn 10: ~180k cumulative → £0.022
Turn 20: ~320k cumulative → £0.038

Total: £0.038 for 20 turns (vs £0.96 for 10 turns without RAG)
Conversation can continue 20-30+ turns
Developer experience: Natural pair programming flow
```

**Annual economics (100 services, realistic workload):**

```
Assumptions:
- 100 services
- 10 changes per service per month = 1,200 changes/month = 14,400/year
- Average 4 turns per change (iterate, refine, test, finalize)
- Total: 57,600 LLM requests per year

Without RAG (full service folders):
57,600 requests × £0.12/request average = £6,912/year

With Service RAG:
57,600 requests × £0.008/request average = £460.80/year

Annual saving: £6,451/year (93% reduction)

Note: This excludes the cost of architectural cleanup, integration fixes,
and review overhead which add another £200k-400k/year without RAG.
```

**Benefit 3: Conversation Continuity**

This is **qualitative**, not just quantitative:

**Without RAG:**
- Context window fills in 4-5 turns
- Must reset conversation frequently
- Loses thread of reasoning
- "Explain then refine" workflows broken
- Frustrating developer experience

**With RAG:**
- Conversations continue 20+ turns naturally
- Context stays relevant and focused
- LLM maintains coherent reasoning
- True pair programming becomes viable
- Developers actually want to use it

**This drives adoption**: Better experience = higher usage = more value realized.

### 3.3 How RAG Retrieval Works (Technical Overview)

**The RAG index structure:**

```
rag/
├── payments.index           ← Service RAG
│   ├── chunks/              ← Code/doc chunks with metadata
│   ├── embeddings/          ← Vector representations
│   └── metadata.json        ← Service scope, update timestamp
├── notifications.index      ← Separate index per service
└── domain/
    └── finance.index        ← Domain-level RAG for strategic questions
```

**Chunking strategy:**

Each service is split into semantic chunks:
- Functions/methods (~200-500 tokens each)
- Test cases (complete test as one chunk)
- Documentation sections
- ADRs (architectural decision records)
- Configuration snippets

**Chunk metadata (enables smart retrieval):**

```json
{
  "id": "payments-refund-domain-service",
  "path": "services/payments/src/domain/RefundService.ts",
  "type": "domain-logic",
  "service": "payments",
  "layer": "domain",
  "concepts": ["refund", "settlement", "payment", "aggregate"],
  "test_coverage": true,
  "last_modified": "2026-01-15"
}
```

**Query → Retrieval flow:**

```
1. Developer request: "Add partial refund support"
   
2. Service identified: payments-service
   
3. RAG query generated:
   - Search terms: ["partial refund", "refund settlement", "refund logic"]
   - Filters: service=payments, type IN [domain-logic, test, adr]
   
4. Vector search returns top 15 chunks by semantic similarity
   
5. Re-ranking by metadata:
   Priority 1: Tests (encode behavior as spec)
   Priority 2: Domain logic (business rules)
   Priority 3: ADRs (explains "why")
   Priority 4: Application layer (use cases)
   Priority 5: Infrastructure (if needed)
   
6. Final context assembled (~8-12 chunks, 25k tokens):
   ✓ refund.test.ts (business rules)
   ✓ Payment.ts (aggregate root)
   ✓ RefundService.ts (domain service)
   ✓ ProcessRefundUseCase.ts (use case)
   ✓ partial-settlement-adr.md (decision context)
   ✓ IRefundRepository.ts (interface)
   
7. Context + request sent to LLM
```

**Why this works better than loading everything:**

1. **Semantic search** finds relevant code, not just filename matching
2. **Metadata filtering** enforces service boundaries (can't retrieve other services)
3. **Prioritization** ensures tests (ground truth) appear first
4. **Small context** means LLM focuses, doesn't wander
5. **Consistent retrieval** means predictable behavior

### 3.4 RAG Update Strategy

**When does RAG rebuild?**

- **On merge to main**: Changed files re-chunked and re-embedded
- **Nightly**: Full rebuild to catch drift, optimize index
- **On demand**: Manual trigger for major refactors

**Incremental updates (efficient):**

```bash
# Only re-process changed files
git diff --name-only HEAD~1 HEAD | \
  grep "^services/payments/" | \
  xargs python scripts/update-rag-index.py --service payments
```

**What goes into RAG at different phases:**

**Design Phase RAG** (broad, exploratory):
- Service templates/patterns
- Architecture guidelines
- Similar services (for reference)
- Domain glossary
- Contract examples

**Evolution Phase RAG** (narrow, focused):
- Actual service code (domain, application, infrastructure)
- Complete test suite
- Service-specific ADRs
- Service contracts (API specs, event schemas)
- Relevant shared library interfaces (read-only)

**What NEVER goes into service RAG:**
- Other services' implementation code
- Global utilities (unless explicitly allowed)
- Deprecated/experimental code (filtered out)
- Infrastructure code (separate Infrastructure RAG)

**What this tier controls:**

### 4.3 How LLM Uses Domain RAG

**Design Phase Example:**

```
Architect Question: "Should refund logic live in payments domain or finance domain?"

RAG retrieves:
- Business capability map (shows payments owns transactions)
- Regulatory constraint: "Refunds must be auditable with payment"
- Existing context boundaries
- Domain language: "Payment" includes lifecycle (authorize, settle, refund)

LLM analyzes:
"Based on the regulatory requirement for atomic auditability 
and the existing definition of 'Payment' in the ubiquitous language, 
refund logic should remain in the payments bounded context. 
This ensures transactional integrity and audit compliance."

Human architect validates: ✓ Approved
Decision frozen in context map
```

**Evolution Phase Example:**

```
Developer Request: "Add subscription-based refunds to system"

RAG retrieves:
- Payments context definition (scope: all payment types)
- Domain invariants:
  ✓ "All refunds must reference an original payment"
  ✓ "Refund amount cannot exceed original payment"
- Payments context boundaries (owns: payment lifecycle)

LLM validates:
✓ Subscription refunds fit within payments context (payment type variant)
✓ Existing invariants still apply
✓ No new cross-context dependencies needed

Proceeds to Tier 2 (Microservice contracts)
```

### 4.4 Verification - Domain Rule Tests

**Domain tests encode business invariants:**

```typescript
// tests/domain-rules/payment-invariants.test.ts
describe('Payment Domain Invariants', () => {
  describe('Refund Rules', () => {
    it('refund cannot exceed original payment amount', () => {
      const payment = Payment.create({ amount: Money.pounds(100) });
      const attemptRefund = () => payment.refund(Money.pounds(150));
      
      expect(attemptRefund).toThrow(RefundExceedsPaymentError);
    });
    
    it('refund must reference original payment', () => {
      const createOrphanRefund = () => Refund.create({
        amount: Money.pounds(50)
        // No paymentId - should fail
      });
      
      expect(createOrphanRefund).toThrow(InvalidRefundError);
    });
    
    it('partial refunds maintain settlement integrity', () => {
      const payment = Payment.create({ amount: Money.pounds(100) });
      payment.refund(Money.pounds(30));
      payment.refund(Money.pounds(40));
      
      expect(payment.totalRefunded).toEqual(Money.pounds(70));
      expect(payment.remainingRefundable).toEqual(Money.pounds(30));
      expect(payment.refundHistory).toHaveLength(2);
    });
  });
});
```

**These tests are:**
1. Retrieved by RAG (high priority)
2. Define behavior that code must satisfy
3. Fail if domain invariants violated
4. Act as executable specifications

## 5. Tier 2 - Microservice Architecture (Service Boundaries)

### 5.1 Sphere of Influence

**What this tier controls:**
- Service boundaries (what each service owns)
- Service contracts (APIs, events, commands)
- Inter-service communication patterns
- Data ownership and sovereignty

**Why this tier exists:**
To prevent services from becoming coupled, breaking each other's contracts, or creating unauthorized dependencies.

### 5.3 How LLM Uses Microservice RAG

**Design Phase Example:**

```
Request: "Design the refund API for payments service"

RAG retrieves:
- API design patterns (RESTful conventions)
- Event-driven patterns (when to use events vs. APIs)
- Anti-patterns:
  ✗ "Never make synchronous calls between services"
  ✗ "Never share database schemas"
- OpenAPI template

LLM generates:

# Refund API (OpenAPI 3.0)
POST /api/v1/refunds
Request:
{
  "paymentId": "PAY-12345",
  "amount": { "value": 50.00, "currency": "GBP" },
  "reason": "CUSTOMER_REQUEST"
}

Response (202 Accepted):
{
```

## 8. CI/CD Guardrails - The Enforcement Layer

### 8.1 The Principle

> **Architecture is enforced by automation, not process.**

**Traditional approach:**
```
LLM proposes change → Human review catches boundary violation → Reject → Waste
```

**This approach:**
```
LLM proposes change → CI runs guardrails → Violation detected automatically → Block before human sees it
```

**Result:** Humans review business logic, not architectural compliance.

### 8.2 The Four Layers of Defense

**Layer 1: RAG Scoping (Pre-emptive)**

```
Developer: "Fix refund bug in payments"
↓
System identifies service: payments
↓
RAG scope locked to: services/payments/**, libs/payment-interfaces/**
↓
LLM CANNOT see other services (physically impossible)
↓
95% of problems prevented here
```

**Layer 2: Path Validation (Pre-commit)**

```yaml
# .github/workflows/validate-pr.yml
name: Validate Service Scope

on: [pull_request]

jobs:
  validate-paths:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Determine target service
        id: service
        run: |
          # From PR label or branch name
          SERVICE=$(gh pr view ${{ github.event.pull_request.number}} \
            --json labels --jq '.labels[].name' | grep '^service:' | cut -d: -f2)
          echo "service=$SERVICE" >> $GITHUB_OUTPUT
      
      - name: Validate changed files
        run: |
          SERVICE="${{ steps.service.outputs.service }}"
          ALLOWED=$(jq -r ".services[\"$SERVICE\"].paths | .[]" monorepo.json)
          CHANGED=$(git diff --name-only origin/main...HEAD)
          
          python3 << 'EOF'
          import sys, fnmatch
          
          service = "$SERVICE"
          allowed = """$ALLOWED""".strip().split('\n')
          changed = """$CHANGED""".strip().split('\n')
          
          violations = []
          for file in changed:
              if not any(fnmatch.fnmatch(file, pattern) for pattern in allowed):
                  violations.append(file)
          
          if violations:
              print(f"❌ Files outside {service} scope:")
              for v in violations:
                  print(f"  - {v}")
              sys.exit(1)
          else:
              print(f"✅ All changes within {service} scope")
          EOF
```

## 9. Making It Robust at Scale

### 9.1 The Scaling Challenge

**As microservice estates grow:**

| Services | Manual Review Feasible? | Review Time per Change | Architectural Violations per Month |
|----------|-------------------------|------------------------|-------------------------------------|
| 1-10 | Yes | 15-30 min | 2-5 |
| 10-30 | Challenging | 30-60 min | 10-20 |
| 30-50 | Bottleneck | 60-90 min | 25-40 |
| 50-100 | Breaks down | 90-120 min | 50-80 |
| 100+ | Impossible | N/A | Uncontrolled drift |

**The traditional failure mode:**
- Reviews become rubber stamps (humans overwhelmed)
- Violations slip through
- Architectural debt accumulates
- System becomes unmaintainable

### 9.2 How This Approach Scales

**The key insight:**
> Make violations **impossible**, not **discouraged**.

**Scaling characteristics:**

| Aspect | Traditional | This Approach |
|--------|-------------|---------------|
| Review burden | O(n²) - exponential | O(1) - constant |
| Violation rate | Increases with scale | Stays at ~0% |
| Token costs | Exponential growth | Linear growth |
| Setup cost per service | Low | Medium (RAG + tests) |
| Ongoing cost per service | High (review) | Low (automated) |

**Why this scales linearly:**

1. **Each service has own RAG index**
   - No global bottleneck
   - Services don't interfere with each other
   - New service = add new index, same pattern

2. **Each service has own test suite**
   - Contract tests prevent breaking consumers
   - Domain tests prevent business logic violations
   - No central test coordination needed

3. **Each service has own CI validation**
   - Path validation runs per-service
   - Dependency checks run per-service
   - No waiting for global builds

4. **Automated enforcement, not human review**
   - CI blocks violations automatically
   - Humans review business logic only
   - Review capacity doesn't become bottleneck
