Hi,

I've been thinking more about our conversation regarding the Bloom filter and phonetic encoding approach for the cloud migration. I could tell you were sceptical about whether this could realistically work given the volume of PL/SQL you're dealing with, and that's a completely fair concern. I wanted to share some thoughts I've been working through on exactly that question.

The core premise remains the same as what we discussed: sensitive fields get encrypted with keys that never enter AWS, identifiers get replaced with HMAC hashes using a peppered key held externally, and fuzzy search capability is preserved through phonetic codes, Bloom filters, and locality-sensitive hashing. The question you raised was essentially "that's all well and good for reads and searches, but what about the thousands of PL/SQL procedures that touch this data?"

I've been working through the different patterns that PL/SQL typically uses, and I think the picture is more encouraging than it might first appear. The key insight is that we can use Oracle's own mechanisms—views, synonyms, and INSTEAD OF triggers—to insulate most existing code from the underlying changes.

The basic trick is this: we rename the original table (say, `citizens` becomes `citizens_base`), create a view called `citizens_v` that reads from the base table and decrypts on-the-fly using wrapper functions, then create a synonym called `citizens` pointing to that view. Any existing PL/SQL that does `SELECT * FROM citizens` continues working unchanged. Oracle resolves the synonym to the view, the view calls the decryption functions, and the calling code receives plaintext without knowing anything happened. For writes, INSTEAD OF triggers on the view intercept INSERTs, UPDATEs, and DELETEs, encrypt the incoming data, compute the fuzzy artefacts (phonetic codes, Bloom filters, LSH buckets), and write to the base table. Again, the calling code doesn't need to change.

Where code does need modification is in WHERE clause comparisons on identifiers. A query that says `WHERE nino = :value` has to become `WHERE nino_hmac = hmac_nino(:value)`, because we're no longer storing the NINO itself—we're storing its HMAC. This is a real change, but it's mechanical and searchable. You can grep for every occurrence and update systematically. The good news is that HMAC columns can be indexed just like any other column, so performance for exact lookups remains essentially unchanged.

Let me walk through the more challenging patterns in more detail, because I think this is where your scepticism is well-founded and where we need to be realistic about the work involved.

**ORDER BY on encrypted columns** is a genuine problem. When you have a procedure that returns citizens sorted alphabetically by name, that ORDER BY clause becomes meaningless if the name column contains ciphertext—you'd be sorting by random-looking encrypted strings rather than by actual names. There are three ways to handle this. First, for small result sets (say, under 500 rows), you can simply remove the ORDER BY from the database and sort in the calling application after decryption. The procedure returns unsorted data, the application receives it, and the application sorts in memory. This works, but it assumes the application has the decrypted values to sort on. Second, you can store a precomputed sort key—an additional column containing something sortable like the first letter of the name, or a phonetic code, or even a normalised uppercase version of the name if you're willing to accept that exposure. The PL/SQL then changes from `ORDER BY name` to `ORDER BY name_sort_key`. Third, you can simply change the sort order to something non-sensitive: `ORDER BY registration_date DESC` instead of alphabetical, if the business can accept that. This is a risk trade-off question: how important is alphabetical ordering, and what are you willing to expose to enable it?

**GROUP BY and aggregations** present a similar challenge. If you have a report that counts citizens by address—`SELECT address, COUNT(*) FROM citizens GROUP BY address`—this breaks completely with encryption. Each time you encrypt "123 High Street" you get different ciphertext (because good encryption includes random initialisation vectors), so the database sees them as different values and groups them separately. The options here are to group by something non-sensitive (postcode rather than full address, region code rather than location), to group by a derived hash that's deterministic (an HMAC of the normalised address, which at least groups identical addresses together though it does leak the fact that they're the same), or to accept that certain aggregations need to happen outside the database in a trusted environment that can decrypt the data first. That last option works for batch reporting but is impractical for interactive queries. I'd want to understand what your actual aggregation requirements are before assuming this is a showstopper.

**Date arithmetic and range queries** are hard. A query like `WHERE dob BETWEEN '1990-01-01' AND '2000-12-31'` cannot work on encrypted dates because the database can't compare encrypted values meaningfully. The encrypted version of 1991 doesn't look anything like the encrypted version of 1992, so there's no way to ask "is this between those two?". The practical solution is to store derived attributes alongside the encrypted date. You might have `is_adult` as a boolean flag, or `age_bracket` as an enumeration ('CHILD', 'ADULT', 'SENIOR'), or `birth_decade` as a number. These derived flags get computed and updated by a nightly batch process that has access to the adapter, decrypts dates, works out the flags, and updates them. Your PL/SQL then queries these derived columns instead of doing arithmetic on the date itself. This does mean some queries that currently work on precise dates would need to be reframed in terms of the available buckets—you can't ask "born after 15 March 1995" unless you specifically store that as a flag, but you can ask "born in the 1990s" if you store a decade indicator.

**String functions** are problematic in the same way. UPPER(name), SUBSTR(address, 1, 10), INSTR(address, 'London')—these all operate on plaintext and produce garbage when applied to ciphertext. If you need the uppercase version of a name for display or comparison, you either compute it at read time after decryption (feasible but adds latency), or you store it as a separate encrypted column that was uppercased before encryption. The latter means storing both `name_enc` (the original) and `name_upper_enc` (the uppercase version), which multiplies storage and write overhead. For partial matching like INSTR, the fuzzy search approach using Bloom filters actually handles this quite well—searching for citizens with "London" in their address becomes a Bloom-based query that finds records where "London" appears as a substring, using the same phonetic/LSH/Bloom pipeline that handles fuzzy name matching. It's different from INSTR, but it works.

**CHECK constraints** that validate data format won't work on hashed or encrypted columns. A constraint like `CHECK (REGEXP_LIKE(nino, '^[A-Z]{2}[0-9]{6}[A-Z]$'))` fails immediately because the stored value isn't a NINO anymore—it's an HMAC hash. The solution is to move validation into the INSTEAD OF trigger, where the plaintext is still available before encryption. The trigger checks the format, rejects invalid data, and only then encrypts and stores. The database constraint on the base table can be dropped or changed to validate the format of the encrypted/hashed value instead.

**Foreign key relationships** need to reference the HMAC columns rather than plaintext identifiers. If the benefits table currently has a foreign key from `citizen_id` to the citizens table, that needs to change to reference `citizen_id_hmac` on both sides. The benefits table needs its own `citizen_id_hmac` column, which gets populated through its own INSTEAD OF trigger that hashes the incoming identifier. This cascades the protection model through related tables, which is good for security but means the schema changes aren't isolated to just the main tables.

**Existing triggers** on the original tables need review. Audit triggers that log "citizen name changed from X to Y" are now logging encrypted values unless they're modified to either log non-sensitive context only, or to run in the INSTEAD OF trigger where plaintext is still available. Validation triggers need to move into the view layer. Cascading triggers that update related tables based on identifier values need to work with HMAC versions.

**Dynamic SQL** using EXECUTE IMMEDIATE is probably the hardest pattern to handle systematically. If you have procedures that build queries at runtime—constructing column names or table names from variables—you can't just grep and replace. Each instance needs individual analysis to understand what's being dynamically built and whether it might reference sensitive columns. If a procedure says `WHERE ' || p_column || ' = :val` and p_column could be a sensitive field, the code needs logic to detect that and use the appropriate HMAC wrapper.

**Bulk operations** deserve special attention because they're often the difference between acceptable and unacceptable performance for batch processing. If you have a FORALL loop that inserts 10,000 rows, and each row triggers an INSTEAD OF trigger that makes an individual call to the external adapter for encryption, you've just turned a bulk operation into 10,000 network round-trips. The performance hit would be brutal. The solution is batch APIs on the adapter—you extract all the plaintext values into PL/SQL collections, make one call to the adapter saying "encrypt these 10,000 names", get back an array of 10,000 encrypted values, then do a bulk insert of the pre-encrypted data into the base table, bypassing the trigger for that specific controlled scenario. This requires adapter support for batch operations and some restructuring of the PL/SQL, but it preserves reasonable throughput.

**Logging and exception handling** are easy to overlook but important. If your code has DBMS_OUTPUT.PUT_LINE('Processing citizen: ' || v_name) or exception handlers that log error context including sensitive values, those plaintext strings would be visible in logs stored in AWS. You need to either remove sensitive data from logs, replace it with HMAC identifiers (traceable but not readable), or ensure logging goes to the trusted environment rather than the cloud.

I've put together a more detailed write-up covering these patterns with difficulty ratings, effort estimates, and performance implications for each. Happy to share that if useful, though it might be more detail than you want at this stage.

The broader point I wanted to make is about risk appetite. Everything I've described is predicated on the assumption that the primary risk we're mitigating is confidentiality—specifically, preventing US authorities from compelling AWS to disclose citizen data under the Patriot Act or CLOUD Act. This approach absolutely addresses that: without the keys (which never enter AWS), the data is computationally useless to anyone who gains access to the database, whether through legal compulsion or breach.

What this approach does not prioritise is availability. The trusted adapter becomes a critical dependency. If it's unavailable, you cannot decrypt data for display, you cannot compute HMACs for lookups, and you cannot write new records. This is a genuine operational risk that needs to be factored into the architecture—high availability, redundancy, failover, and so on. The adapter has to be treated as production-critical infrastructure, because it is.

On integrity, the picture is actually somewhat positive. HMAC provides implicit integrity verification—if someone tampers with a stored hash, the comparison will fail. Encrypted fields with authenticated encryption modes provide similar guarantees. So while integrity isn't the primary goal, the approach does contribute to it.

The question for you and the programme is whether these trade-offs make sense given your actual risk profile. If Patriot Act exposure is the existential concern keeping people awake at night, this approach directly addresses it while requiring a manageable (though not trivial) amount of PL/SQL adaptation. If uptime and operational simplicity are the dominant concerns, this approach adds complexity and failure modes that need to be weighed carefully.
