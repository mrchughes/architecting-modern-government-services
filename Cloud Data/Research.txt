Got it — Bloom filters are a bit trickier conceptually, so let’s break it down step by step, with a concrete example.

1. What is a Bloom filter?
A Bloom filter is a bit array (like a string of 0s and 1s) that represents whether elements are probably in a set.

It never stores the original data.

It can give false positives (says an element exists when it doesn’t), but never false negatives (won’t miss something that’s actually there).

You can use it to compare similarity between strings without storing plaintext.

2. How it works for fuzzy matching
Let’s take a simple example:

Step 1: Break your string into n-grams
Example: name = "John Smith"

Generate 3-grams (groups of 3 consecutive letters):

"Joh", "ohn", "hn ", "n S", " Sm", "Smi", "mit", "ith"
Step 2: Hash each n-gram
Use one or more hash functions (could be HMAC with a secret key)

Map the hash to a position in a fixed-size bit array

Bit array size = 16 (for simplicity, normally much bigger)
"Joh" → hash → position 3 → set bit 3 = 1
"ohn" → hash → position 7 → set bit 7 = 1
...
Step 3: Store the bit array
You never store the original name, only the bit array.

Example:

[0,0,0,1,0,0,0,1,0,0,0,1,0,0,0,1]
3. Query / fuzzy match
Suppose you want to match "Jon Smith":

Generate 3-grams:

"Jon", "on ", "n S", " Sm", "Smi", "mit", "ith"
Hash each n-gram and map to bit positions

Compare number of bits that overlap with stored Bloom filter

The more overlapping bits, the more likely the strings are similar → this gives a fuzzy match score

No plaintext is exposed to the cloud

4. Why Bloom filters help with privacy
Even if someone sees the bit array, they can’t reconstruct the original name easily.

Combined with a secret hash key (HMAC), it becomes nearly impossible to reverse.

Supports typos and minor differences, unlike deterministic hashing.

5. Quick analogy
Think of it like a fingerprint of the string:

Exact hash → fingerprint of entire hand → no similarity detection

Bloom filter → fingerprint of multiple features (n-grams) → can see “roughly similar” fingerprints

what defines n in n-grams

Ah — this is a really important part of the design, because the choice of n affects how fuzzy your matching is. Let’s go through it clearly.

1. What an n-gram is
An n-gram is simply a consecutive sequence of n characters (or words) from a string.

n = 1 → 1-grams (single letters)

"John" → J, o, h, n

n = 2 → 2-grams (bigrams)

"John" → Jo, oh, hn

n = 3 → 3-grams (trigrams)

"John" → Joh, ohn

n = 4 → 4-grams (quadgrams)

"John" → John

2. How n affects fuzzy matching
n	Effect	Pros	Cons
1-2	Very fine-grained	Can detect small typos, insertions, deletions	Lots of noise; many false positives
3	Common default	Good balance between typo tolerance and noise	Slightly less sensitive to single-character errors
4+	Coarser	Better for exact matches, less false positives	Misses small edits or spelling variations
Example typo: "John Smith" vs "Jon Smith"

n	Overlap
2-grams	"Jo", "oh", "hn" vs "Jo", "on", "n " → some overlap → fuzzy match works
3-grams	"Joh", "ohn" vs "Jon", "on " → less overlap → still matches moderately
4-grams	"John", "ohn " vs "Jon ", "on S" → almost no overlap → fuzzy match fails
So smaller n = more sensitive to typos, larger n = stricter match.

3. Typical choices
Names → 2–3 grams are typical

Addresses → 3–4 grams (because longer strings, less false positives)

Free-text fields → sometimes 3–5 grams

4. Trade-offs in Bloom filter context
Smaller n → more n-grams → more bits set → higher false positive chance

Larger n → fewer n-grams → may miss matches on small typos

Usually, you pick n empirically by testing on your dataset to balance match sensitivity vs false positives.

Ah — this is a common point of confusion, because they both use “hashes”, but the purpose and behavior are very different. Let’s break it down carefully.

1. Hash Index (Exact-Match)
Purpose: Find exact matches of a key (like NINO or hashed ID) quickly.

How it works:

Compute a hash of the key → maps to a bucket.

Bucket contains row pointers that share that hash (usually very few due to collisions).

Compare only rows in that bucket → find exact match.

Characteristics:

Deterministic: same input → same hash → same bucket

Works only for exact matches

Lookup time ~ O(1) → constant, independent of table size

Example: hash(NINO) → bucket → find exactly John Smith with that NINO

2. Locality-Sensitive Hashing (LSH) (Approximate / Fuzzy-Match)
Purpose: Find approximate / similar matches quickly (e.g., names that are similar, not identical).

How it works:

Take a fingerprint of the record (e.g., Bloom filter of n-grams of the name).

LSH hashes similar fingerprints into the same bucket with high probability.

When querying, compute fingerprint of query → find LSH bucket → only compare rows in that bucket.

Characteristics:

Probabilistic: similar inputs → usually same bucket, but not guaranteed

Works for approximate / fuzzy matches

Lookup time is very fast → only need to check small subset of candidates instead of millions

Example: "Alice Smith" → fingerprint → LSH bucket 7

"Alicia Smyth" → similar fingerprint → also likely in bucket 7

Only bucket 7 compared for fuzzy match

3. Key Differences
Feature	Hash Index	LSH (Locality-Sensitive Hashing)
Match type	Exact	Approximate / fuzzy
Input → bucket mapping	Deterministic	Probabilistic (similar inputs → same bucket)
Lookup guarantee	Always finds exact match	Likely to find similar items; may miss very rare cases
Purpose	Fast exact-match search	Reduce comparisons for fuzzy search
Hash function	Standard cryptographic / non-cryptographic hash	Special LSH functions designed to group similar fingerprints
4. Analogy (without over-simplifying)
Hash index: like a lock and key — only the exact key opens the lock.

LSH: like a neighborhood — keys that are “close” live in the same neighborhood, so you only have to check that area for approximate matches.

✅ Bottom line:

Hash index = exact match → you always know the key you’re looking for

LSH = fuzzy / approximate match → narrows search to likely candidates for similarity

o got these comments: This is the architecture doc from your cloud sovereignty work. A few thoughts:

What it does well

The layered approach is sound — phonetic partitioning as a coarse filter, LSH as a medium filter, Bloom filter comparison as fine-grained, then local decryption for final verification. Each layer reduces the candidate set without touching plaintext. The separation between "things we hash" (relationships, exact-match IDs) and "things we encrypt" (NINO, DOB, addresses) is the right distinction and is clearly explained.

The executive summary is genuinely readable for a non-technical audience, which matters for getting deputy director sign-off.

Where I'd push back or sharpen

The relationship hashing section is the weakest link and deserves more scrutiny. If you hash a relationship ID like a NINO or citizen ID, and the attacker knows the ID space (which for government systems is often constrained), they can brute-force the hash. You either need a keyed HMAC with a locally-held key, or you accept that relationship graphs are pseudonymous rather than properly protected. The doc elides this.

The Bloom filter collision/false positive rate isn't addressed quantitatively. For a production system at DWP scale (tens of millions of citizens), you'd want to specify the filter size and hash function count to keep false positive rates acceptable — otherwise the local verification step becomes a bottleneck.

The update story is undercooked. "Can be batched efficiently" is doing a lot of work. If a citizen changes their name, you're recomputing n-grams, Bloom filters, phonetic codes, and LSH buckets, then pushing all of that. Salt is typically unique per record and stored alongside the hash. It prevents attackers from using precomputed rainbow tables, but the salt itself must be stored (usually in the cloud). Locality-Sensitive Hashing (LSH):
LSH is used to reduce the number of records that need to be compared during fuzzy search. Instead of hashing the entire Bloom filter (which would only support exact matches), the Bloom filter is split into smaller parts (“bands”), and each part is hashed separately. This means that similar names, which produce similar Bloom filters, are likely to share at least one band and therefore fall into the same bucket. At query time, only records in matching buckets are retrieved and compared using Bloom filter overlap, significantly reducing the number of comparisons while preserving fuzzy matching capability.

def bloom_overlap_score(bloom1, bloom2):
    # compute Jaccard similarity: (# of shared 1s) / (# of 1s in union)
    overlap = sum(bit1 & bit2 for bit1, bit2 in zip(bloom1, bloom2))
    union = sum(bit1 | bit2 for bit1, bit2 in zip(bloom1, bloom2))
    return overlap / union

    if score >= 0.7:  # similarity threshold

why use more than one hash?

Good question — this is a classic Bloom filter design point.

Here’s why you usually use multiple hash functions per n-gram instead of just one:

1️⃣ Reduce false positives
One hash per n-gram only sets one bit in the array.

Two different strings with a few n-grams may accidentally collide in that single bit → false match.

Multiple hashes set several bits per n-gram → reduces chance of random collisions.

Example (16-bit array):

n-gram	hash1	hash2	hash3	Bits set
Al	1	7	12	1,7,12
li	2	5	10	2,5,10
More bits = more “signature” of the n-gram

Only strings that actually share most of the same n-grams will have high overlap

2️⃣ Increase Bloom filter “information content”
One bit per n-gram → very coarse → many different names may look similar

Multiple bits per n-gram → the filter encodes more detail of the string

This makes your fuzzy matching more accurate

3️⃣ Trade-off
Pros of more hashes: lower false positives, better discrimination

Cons: slightly more computation per n-gram, slightly higher storage (since more bits get set)

In practice: 2–5 hashes per n-gram is common for names, 64–256 bit filters

💡 Mental shortcut:

Multiple hashes = multiple “checkpoints” per n-gram → harder for two different strings to look identical by accident.

hmac just peppered hash?

Not exactly — HMAC and a peppered hash are related concepts, but there’s an important distinction:

1️⃣ Peppered hash
You take a normal hash of a value (e.g., NINO) and add a “pepper” — a secret value stored locally — before hashing:

hash_value = hash(nino + pepper)
Only someone with the pepper can compute or verify the hash.

Simple, works for exact matches, but usually uses one hash function.

2️⃣ HMAC (Hash-based Message Authentication Code)
HMAC is a standard cryptographic construction:

It combines a secret key with the value in a structured way to produce a hash

Resistant to known attacks that might work on simple peppered hashes

Formula (simplified):

hmac = hash_function(key ⊕ opad || hash_function(key ⊕ ipad || message))
Properties:

Deterministic: same input + key → same output

Keyed: without the key, an attacker cannot brute-force effectively

Widely vetted: cryptographically secure, unlike ad-hoc peppering

So, the difference
Feature	Peppered Hash	HMAC
Uses secret key	Yes (pepper)	Yes (cryptographic key)
Security rigor	Informal, ad-hoc	Standardized, cryptographically secure
Resistance to attacks	Moderate	Very high
Use case	Exact-match IDs, small input spaces	Exact-match IDs, secure messaging, small input spaces
TL;DR: A peppered hash is a quick-and-dirty version of a keyed hash; HMAC is a secure, standard way to do the same thing but much safer, especially for sensitive identifiers like NINO or citizen IDs.

Proposal: Secure Cloud Migration of Citizen Data with PL/SQL Preservation
Overview
We propose a migration of the existing citizen database to AWS that materially reduces the risk of unauthorised access to sensitive data (including risks associated with extraterritorial legislation such as the USA PATRIOT Act), while preserving the current PL/SQL estate and avoiding a multi-year rewrite.

The core idea is simple but powerful: the cloud platform will store and process only protected representations of sensitive data, and will never hold the keys required to interpret that data. The ability to convert protected data back into intelligible form is isolated in a tightly controlled “trusted environment” outside the AWS trust boundary.

This allows us to retain the operational and scaling benefits of cloud infrastructure, while significantly reducing the exposure of citizen data.

How the Model Works in Practice
At a high level, the system separates data into three categories based on how it is used: values that must be read, values that must be matched exactly, and values that must be matched approximately.

Fields such as name, date of birth and address are stored in encrypted form. These are only ever decrypted when absolutely necessary, and that decryption is performed outside AWS by a dedicated adapter service that holds the keys.

Identifiers such as NINO and internal citizen IDs are not stored in a readable form at all. Instead, they are transformed using a keyed HMAC. This produces a consistent value for matching and joining, but cannot be reversed or guessed without access to the key. This allows the database to perform joins and exact lookups without ever seeing the underlying identifier.

For fuzzy matching, which is essential for identity resolution at scale, the system stores non-reversible derived data. Names are broken into n-grams, these are encoded into Bloom filters (bit arrays representing the presence of substrings), and those are further indexed using locality-sensitive hashing (LSH). These structures allow the database to efficiently identify similar records without exposing the original text.

The important point is that none of these representations—HMACs, Bloom filters, or LSH buckets—can be used to reconstruct the original data.

End-to-End Behaviour
From a user or application perspective, the system behaves much as it does today, but internally the execution path is layered.

A typical fuzzy search starts by transforming the input into the same derived forms used in storage. The database first narrows the search using phonetic matching, then further reduces the candidate set using LSH bucket overlap, and finally applies Bloom filter similarity to identify likely matches. At this point, the result set has typically been reduced from millions of records to a few dozen or fewer.

Only at this final stage are records passed to the trusted environment, where decryption occurs and exact comparisons can be made. This ensures that sensitive operations are both rare and tightly controlled.

For exact lookups—such as retrieving a citizen by NINO—the process is even simpler. The input is converted to an HMAC using the external key, and the database performs a direct equality match. No decryption is required unless data is being displayed.

CRUD operations follow the same pattern. On creation or update, sensitive fields are encrypted and transformed before being written to the database. On read, only the required fields are decrypted, and only after the dataset has been reduced to a minimal set.

Preserving Existing PL/SQL
A key requirement is to avoid rewriting the existing PL/SQL estate. This is achieved not by changing the logic, but by introducing a controlled indirection layer.

Tables are wrapped with views that expose decrypted values via wrapper functions. These functions call out to the trusted adapter to perform decryption or HMAC generation as required. In many cases, existing PL/SQL can continue to operate with only minor adjustments, such as pointing to a view instead of a base table or using a wrapper function for key comparisons.

This approach avoids wholesale refactoring while still enforcing the new security model.

It is important to note that not all patterns are equally efficient under this model. PL/SQL that performs large-scale iteration over sensitive fields may incur performance penalties due to repeated external calls. However, the design deliberately minimises such cases by ensuring that filtering happens before decryption.

Migration Approach
Migration is designed to be incremental and low-risk.

Initially, new protected columns are introduced alongside existing plaintext columns. Data is backfilled in batches, with encryption, HMAC generation and fuzzy artefact computation performed offline. During this phase, the system continues to operate as normal.

A dual-write phase follows, where new and updated records are written in both formats. This allows validation of the new model without disrupting existing functionality.

Once confidence is established, read paths are switched to use the protected representations via views and wrapper functions. Only after full validation are the original plaintext columns removed.

This phased approach avoids a “big bang” migration and allows issues to be identified early.

Security Position
This design significantly reduces the attack surface of the system.

Sensitive data is never stored in plaintext in the cloud, and the keys required to decrypt or generate identifiers are never present in AWS. Even in the event of full database compromise, an attacker would obtain only encrypted values and non-reversible transformations.

The use of HMAC for identifiers addresses a common weakness in pseudonymisation schemes. Because the transformation is keyed, it is resistant to brute-force attacks even when the input space is known or constrained.

The trusted environment becomes the critical control point and must be treated accordingly. It should be isolated, hardened, and ideally backed by hardware security modules. The definition of “trusted” in a distributed estate (e.g. multiple data centres, failover environments) will need to be explicitly agreed as part of the broader architecture.

It is important to be clear that this approach does not eliminate all risk. It does, however, move the system from a model where cloud access equates to data access, to one where meaningful access requires compromise of a separate, tightly controlled environment.

Performance Characteristics
The design deliberately separates high-volume operations from sensitive operations.

Most query processing—phonetic filtering, LSH lookup, Bloom comparison, and HMAC matching—is performed entirely within the database and scales well using standard indexing techniques.

Decryption is comparatively expensive, as it requires a call to the external adapter. However, because it is only performed on a very small subset of records, the overall impact is manageable. In practice, even at national scale, the number of records requiring decryption per query is expected to be in the tens rather than thousands.

The main performance risks arise where this discipline is not maintained, for example if PL/SQL performs decryption inside large loops or before filtering. These cases will need to be identified and optimised.

Cost Considerations
There are additional costs associated with this architecture.

There is a compute overhead for generating and storing fuzzy artefacts, and additional storage overhead for maintaining multiple representations of the same logical field. The trusted adapter layer introduces both infrastructure cost and operational overhead, particularly if deployed in a highly available, multi-region configuration.

There is also an indirect cost in increased complexity. Development, debugging and operational support will all require a deeper understanding of the data transformations and execution flow.

However, these costs should be weighed against the avoided cost of a full system rewrite, and the risk reduction achieved in protecting sensitive citizen data.

Key Risks and Issues
The most significant risk is the introduction of the trusted adapter as a critical dependency. If it is unavailable or degraded, operations requiring decryption or HMAC generation will be impacted. This must be mitigated through robust design, including redundancy, caching where appropriate, and clear fallback strategies.

There is also a risk that the definition of the trusted environment becomes blurred in practice. If keys or decryption capabilities are inadvertently extended into the cloud environment, the security model is undermined. Strong governance and clear architectural boundaries are essential.

From a data perspective, fuzzy matching always carries a risk of false positives and false negatives. While the layered approach (phonetic, LSH, Bloom, exact verification) mitigates this, thresholds and parameters must be carefully tuned and validated at scale.

Finally, while the approach preserves most PL/SQL, it does not guarantee zero change. Some edge cases will require adjustment, and performance tuning will be necessary in parts of the estate.

Conclusion
This proposal offers a pragmatic path to cloud adoption that balances security, cost, and delivery risk.

It avoids the extremes of either accepting full cloud trust or undertaking a complete system rewrite. Instead, it introduces a controlled separation between data storage and data interpretation, allowing the organisation to retain its existing investment in PL/SQL while materially improving the protection of sensitive citizen data.

With careful implementation and governance, this approach provides a credible and defensible foundation for handling citizen data in a cloud environment.
